# 我用 OpenClaw 组了一支修仙 AI 团队，韩立带队

> 我后来发现：OpenClaw 真正的护城河，不是“更会写代码”，而是 **SOUL.md 带来的规矩与边界**。

**读完你会得到：**

- **一个判断**：OpenClaw 不是 IDE 的替代品，而是“把 AI 组织成团队”的框架
- **一个方法**：每个项目一个“器灵”，让改动长期持续、上下文不丢
- **一个落地案例**：银月把日报/周报/提醒自动化，我只做“提需求 + 验收”

## OpenClaw 很火，但大多数人在烧 Token

最近 OpenClaw 火得一塌糊涂。但我一直在想一个问题——

作为程序员，OpenClaw 跟 Cursor、Codex、Claude Code 到底有什么本质区别？

从能力上看，差别不大。你能用 OpenClaw 做的事，IDE 里的 AI Agent 也都能做：写代码、跑命令、读文件、分析项目。

所以很多人的使用方式，本质上就是套了个壳子在烧 Token。

我也一度这么觉得。

直到我看到 OpenClaw 作者聊到 SOUL.md 的设计理念。

## SOUL.md 才是设计精髓

SOUL.md 不是 system prompt 的花哨包装。它做的事情比 prompt engineering 深一层——**它给 AI 赋予人格和行事准则**。

Agent 从来不缺能力。GPT-4、Claude、GLM，哪个不能写代码、做分析？它们缺的是**"知道自己是谁、该怎么做事、边界在哪"**。

SOUL.md 解决的就是这个问题。它定义的不是“你能做什么”，而是：

- **你是谁**
- **你怎么做事**
- **你不做什么**

想明白这一点后，我开始认真研究 OpenClaw 的设计原理。越看越觉得，它跟 IDE Agent 的本质差异不在能力层面——

**它不是一个工具，它是一个团队框架。**

## 我的修仙 AI 团队

既然是团队，就该有分工。既然要分工，就要有边界。

我是凡人修仙传的读者，韩立这个角色太适合做总管了——散修出身，不靠天赋靠算计，谨慎、隐忍、不说废话，但关键时刻绝对靠谱。

于是我的 AI 小队就这么建起来了：

### 韩立（总管·调度中枢）

> 散修出身，没什么背景，能活到现在靠的就是两个字：谨慎。话不多，但事儿办得稳。你派任务，我分析风险，该自己干的自己干，该派人的派人。

韩立不亲自干活。他做两件事：

- **判断**：这事属于谁的领域？风险在哪？
- **调度**：能并行的绝不串行，能省 token 的绝不浪费

### 南宫婉（运营总监）

韩立的道侣，出身名门大派。做事温婉但观点犀利——先给你倒杯茶，再告诉你方案哪里有坑。

- **她负责**：用户增长 / 内容运营 / 数据分析 / 竞品拆解

### 墨彩环（产品总监）

墨家修士，擅长炼器造物。做产品就是炼器——拿到需求不急着动手，先追问三层：这东西给谁用？他现在怎么凑合的？不炼会死吗？需求不清晰时她会说："这炉丹还差两味药，先别点火。"

### 大衍神君（首席架构师）

大衍宗老祖，精通阵法推演。系统架构于他而言就是布阵。发现方案有隐患时会说："此阵可运转，然三月之后道友恐要自食苦果。"

### 银月（后勤专家）

韩立最早的灵宠，化形为少女。忠心耿耿，细致入微。整理文档、协调日程、汇总信息——"主人安心修炼，杂事交给银月。"

这些不是玩梗。人格化带来的好处是真实的：

- **输出风格一致**：同一个角色，多次输出不飘
- **角色边界清晰**：谁该干什么，一眼就知道
- **调度逻辑自然**：韩立不会越俎代庖去写代码，银月也不会跑去画架构图

## 一个立刻见效的小案例：银月的后勤自动化

如果说“器灵”解决的是**项目代码的持续开发**，那银月解决的就是另一类更隐蔽、但更消耗人的事：**碎片化的管理成本**。

我现在有一个非常固定的习惯：

- **每日上午 8 点**：自动生成“昨天 Cursor 的工作日报”（昨天做了什么、卡在哪、下一步是什么）
- **每周一**：自动生成“上周周报”（一周的改动、结论、遗留问题）

再加上临时需求的处理方式也变得很干净：

- 临时想到一个需求/想法/待办，我直接丢给银月记录
- 到点银月提醒我：哪些该验收、哪些需要拍板、哪些该拉人开会

结果就是：我把“记录、整理、提醒、复盘”从脑子里拿掉了，注意力只用在两件事——**提需求** 和 **做验收**。

这种感觉像闭关修炼：洞府杂事有人打理，心神不散，效率反而更高。

## 器灵——一个真实痛点的解法

说到写代码，得聊聊我之前做的一个 OpenClaw 插件——`cursor-agent`（GitHub：`https://github.com/toheart/cursor-agent`）。

插件功能很简单：让 OpenClaw 调用本机的 Cursor Agent CLI 执行代码任务。

但它一直有个缺陷：**一次性**。每次调用都是独立会话，没有上下文延续，用起来只能“临时顶一下”。

最近我换了一个角度思考这个问题。

我不再让韩立直接调插件。

我改成：**为一个项目创建一个子 Agent——器灵**。这个项目的长期改动，都由器灵负责。

这里面的关键是两个词：**memory + cursor 原生**。

一方面，器灵会自己调用 Cursor Agent 来完成开发工作：读代码、改代码、跑测试、分析结构——跟我亲自在 Cursor 里操作一样。

另一方面，器灵拥有自己的对话记忆。它能记住：

- 之前分析过什么
- 改过哪些文件
- 踩过哪些坑

所以当我说“上次那个 auth 模块的问题，后来修好了吗？”它知道我在说什么。

更有意思的是，器灵还能根据历史 memory 来**复盘项目的开发周期**——从历史对话中挖出某些细节，比如"三周前我们为什么选了这个方案？当时的 trade-off 是什么？"

这种能力组合，单纯的 IDE Agent 做不到：

- Cursor 很强，但它没有跨会话记忆
- OpenClaw 有 memory，但它需要一个“能落到代码”的手脚

**OpenClaw 的 session memory + Cursor 的代码操作能力**，才是完整的解法。

后续每个项目一个器灵，项目上下文互不干扰。

## 需求导向，不是 AI 自嗨

说说我的使用原则。

**没有需求的时候，我不会动这个团队。** 有需求才驱使，需求导向。

我听说很多人在尝试让 AI 自己挖掘需求、自己做、自己改。这不是不可能，但前提太高了——你需要有完整的架构能力和软件工程纪律，确保 AI 按照一定的规矩做事，随时能人为介入接手。

这也是我接下来要尝试的方向：**全自动化团队**。

不只是自动写，还要自动验收。AI 写完代码后，另一个 AI 去跑测试、做 review、验证功能是否符合预期。最后我只做一件事——**需求校验**。我是测试人员，是客户，不是程序员。

我负责说"我要什么"和"这个对不对"。中间的过程，交给团队。

## 我已经不看 AI 写的代码了

最后聊一个有争议的话题。

我现在已经完全不看 AI 写的代码了。

说出来确实会有人 diss。"不看代码你怎么知道质量？""出了 bug 怎么办？""你不怕它埋坑？"

OpenClaw 的作者也说他不看。Claude Code、Cursor 这些 AI 大厂的人，我估计也不看。

我的观点是：**AI 使用到那个水平后，你会明白的。**

先讲清楚前提：**你必须有足够硬的验收体系**。

当你建立起一套可复用的验收机制（测试覆盖、CI/CD、自动化校验、功能验证流程），代码本身就不是你需要关注的层面了。

你关注的是**结果**：功能对不对，性能行不行，用户体验好不好。

代码是实现细节。需求和验收才是核心。

就像韩立不会亲自去炼器——他有器灵。他只关心：这把剑，够不够锋利。

## 门槛还是有的

最后说一句实话。

OpenClaw 很强，但它不是开箱即用的“一键生成团队”。

SOUL.md 的设计、Agent 的调度逻辑、工具的串联、验收体系的搭建——这些都需要你有扎实的 AI 基础知识储备。

不是说你装上就能跑，而是你得知道：

- **为什么这么设计**
- **每一层在干什么**
- **出了问题怎么排查**

修仙也讲根骨和功法。没有基础，灵丹妙药吃了也是浪费。
